# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Kiriha is a Windows desktop file manager (Avalonia UI, .NET 10, native AOT) that intentionally combines two UI vocabularies: **Chrome-inspired vertical tabs / bookmarks / settings**, and **Windows Explorer's** navigation and file behavior (breadcrumb address bar, command bar, shell context menus, quick access, clipboard/DnD semantics). When implementing a feature, the default is "match what Explorer does" unless the feature is inherently tab/browser-shaped (in which case "match what Chrome does").

The desktop app is the only .NET project: `src/Kiriha/Kiriha.csproj` (no test project, no CI config). The repository also contains the local signed-release pipeline in `scripts/release-local.ps1` and the Cloudflare Worker / landing page in `web/`.

## Build & run

```bash
dotnet build src/Kiriha/Kiriha.csproj      # or: dotnet build Kiriha.slnx
dotnet run --project src/Kiriha/Kiriha.csproj
```

Native AOT publish:

```bash
dotnet publish src/Kiriha/Kiriha.csproj -c Release
# output: src/Kiriha/bin/Release/win-x64/publish/Kiriha.exe
```

From Git Bash, AOT's link step needs vswhere on PATH or it fails:
```bash
export PATH="$PATH:/c/Program Files (x86)/Microsoft Visual Studio/Installer"
```

When an explicit release is requested, use the `vava` workflow: `vava.config.json` invokes `scripts/release-local.ps1`, which publishes and signs the x64/ARM64 Velopack artifacts, uploads them to Cloudflare R2, verifies delivery, and prunes obsolete packages. The script has external effects and reads the Cloudflare credential from `C:\Users\IMT\dev\Secret\secrets.json`; do not use it as a normal build or verification command. `web/worker.js` serves both the landing page and R2-hosted update files through `kiriha.nephilim.jp`.

`TreatWarningsAsErrors` is on (`Directory.Build.props`). `IL3053` is suppressed in the csproj — it's a known false-positive from Velopack's COM interop that ILC misreads as invalid IL; don't "fix" it, it's already handled.

`RestorePackagesWithLockFile` is on, so `src/Kiriha/packages.lock.json` is committed and `dotnet restore` validates against it. Changing a `PackageReference` regenerates the lock file — commit that change together with the csproj edit rather than deleting the lock file.

There are no automated tests. Verification is: build clean (0 warnings), run the Debug exe, and screenshot/exercise the actual UI for anything touching XAML or interaction — a green build does not mean a feature works.

## Architecture

**MVVM, one window, tab-scoped state.** `MainWindowViewModel` owns the vertical tab collection, sidebar, bookmarks bar, and window-level settings. Each open tab is an independent `TabViewModel` with its own path, navigation history (`_back`/`_forward` stacks), sort/view state, search/filter, and selection. Folder-change watching is **not** per-tab: `TabViewModel` holds only an `IDisposable` subscription handle and delegates to `DirectoryObservationService` (Services/DirectoryObservationService.cs), which keys one shared `FileSystemWatcher` per normalized Windows path across all tabs (reference-counted, with a debounce) so opening the same folder in multiple tabs doesn't spawn duplicate watchers. The **Settings tab** (`chrome://settings`-style) is just a `TabViewModel` constructed with `isSettingsTab: true` — `IsSettingsTab`/`IsNormalTab` gate which half of `MainWindow.axaml`'s per-tab `DataTemplate` renders. There is no separate settings window.

**Shared state that must stay shared across tabs** lives in a few singleton-ish objects passed into every `TabViewModel`, not duplicated per-tab:
- `ShellOptions` (Services/ShellOptions.cs) — show-hidden / show-extensions / show-checkboxes / icon-set (`FileIconSet`: `Original` emoji, `Material`, `Windows` shell icons), with a `Changed` event every tab subscribes to so each option propagates across all open tabs.
- `ClipboardFileService` — process-wide cut/copy state (`CutStateChanged` event) so every tab's file list can show the Explorer-style "cut item is translucent" effect, not just the tab that did the cutting.
- `AppSettings` (Services/SettingsService.cs) — the JSON-serialized (source-generated, AOT-safe) blob persisted to `%LocalAppData%\Kiriha\settings.json`: pinned tabs, bookmarks tree, window bounds, view-mode default, theme, startup path, etc. `MainWindowViewModel` holds the one `AppSettings` instance and writes through `SettingsService.Save` on every relevant mutation — there's no debounce, so don't be surprised by frequent small writes.

**Licensing is a one-time purchase with a signed-key scheme** (Services/LicenseService.cs, no external license library). State machine: `Licensed` / `Trial` (14 days, start date persisted in both `%LocalAppData%\Kiriha\trial.dat` and `HKCU\Software\Kiriha\TrialStart`, earliest wins) / `TrialExpired` / `OnlineCheckRequired` (a valid key whose online revocation check hasn't succeeded for 30 days). Keys are `KIRIHA-<base64url(payload)>.<base64url(sig)>`, ECDSA P-256-signed and verified offline against public keys embedded in `PublicKeysSpki` (an array to allow key rotation — never change the key format itself; the private key lives outside the repo in `dev\Secret\kiriha-license`). Purchase and revocation are served by the Sekisho hub (`sekisho.nephilim.jp`, external dependency); revocation checks fall back to the `kiriha.nephilim.jp/license/check` compat proxy. **There is deliberately no device/seat limit** — one purchase means one key that works on any number of machines, and the only enforcement is per-purchase revocation (a refund invalidates the key on every machine at its next check). The app sends only the purchase ID, no device identifier, so a limit could not be added on the hub alone; don't add one without an explicit decision to change the product. When `TrialExpired`/`OnlineCheckRequired`, `MainWindowViewModel.IsLicenseLocked` becomes true: `MainWindow.axaml` shows a modal lock overlay **and** the main-content Grid is disabled via `IsEnabled="{Binding !IsLicenseLocked}"` — both are required (the overlay only blocks pointer input; control-attached `KeyDown` handlers like `FileList_KeyDown` fire before `Window.OnKeyDown`'s lock guard, so keyboard paths are only blocked by the `IsEnabled` binding). Keep all three in sync when touching lock behavior.

`Program` enforces a **single process** with a named mutex. Later launches forward up to 32 startup arguments through a current-user-only named pipe, and `App` dispatches them to `MainWindowViewModel.OpenShellPaths` on the UI thread. When changing startup or activation behavior, preserve the pending-activation handoff used before the UI handler is registered. For manual launch testing, make sure the process under test is the first Kiriha instance; otherwise the new process exits after notifying the existing window.

**Windows Explorer parity is implemented via direct Win32/Shell COM interop**, not by shelling out or reimplementing Explorer's behavior in managed code. Read the existing service before adding a new interop point — the pattern is `internal static partial class` + `[LibraryImport]` (source-generated P/Invoke, AOT-compatible) or source-generated COM (`[GeneratedComInterface]`) rather than classic `[ComImport]`/`DllImport`:
- `ShellContextMenuService` — real `IContextMenu`/`IShellFolder` so right-click gives the actual OS verbs (copy, delete, properties, "Open with", extension-installed entries, etc.), not a reimplementation. Note the deliberate scope decision recorded in comments: Windows 11's redesigned context menu is Explorer-internal with no public API, so this shows the classic/full menu instead — that's expected, not a bug to "fix" toward the new visual style.
- `ClipboardFileService` — writes real `CF_HDROP` + `Preferred DropEffect` clipboard formats so Ctrl+X/C/V round-trips with actual Explorer windows.
- `FileOperationService` — serializes calls to `SHFileOperationW` so delete goes to the Recycle Bin and copy/move/delete get the system progress UI instead of raw `File.*` operations. Known constraint: `SHFileOperationW` is a legacy API with an inherent MAX_PATH (260) limit on `pFrom`/`pTo`, so operations on very deep/long paths can fail (the error is surfaced to the user in Japanese via `DescribeError`). Migrating to `IFileOperation` COM would lift this but is a large rewrite — deferred until it actually bites.
- `QuickAccessService` — reads the real Quick Access shell folder via COM so the sidebar matches the user's actual pinned folders, not a hardcoded guess. It enumerates on a dedicated **STA** thread: that shell namespace extension needs STA, and calling it from the `Task.Run` thread pool (MTA) fails intermittently — the symptom is the sidebar silently falling back to the six standard folders, seemingly at random. `CoInitializeEx` on an MTA thread was tried and was **not** sufficient; keep the STA thread.
- `ShellNewService` — reads the `HKCR\...\ShellNew` registry entries so "New > ..." matches what Explorer would offer on this machine.
- `WindowsIntegrationService` — owns the user-scoped HKCU startup entry, Explorer "Kiriha で開く" verbs, and default folder/drive open association. If registrations change, keep `Program`'s Velopack uninstall callback in sync so uninstall removes or restores everything Kiriha changed.
- `ImageMetadataService` — reads Exif/file properties through `IPropertyStore` (source-generated COM) and lets propsys format the values; see the gallery section below.
- `ShareService` — opens the Windows share sheet via WinRT `DataTransferManager` (`IDataTransferManagerInterop` + CsWinRT projections; needs the `net10.0-windows10.0.26100.0` TFM and the `[GeneratedWinRTExposedExternalType]` assembly attribute for the AOT CCW). Reference this when adding new WinRT interop points.

**File list rendering is three parallel `ListBox`es, not one.** `MainWindow.axaml` has separate `ListBox`es for Details / List / Icons view, all bound to the same `TabViewModel.Entries`, toggled via `IsVisible` bindings on `IsDetailsView`/`IsListView`/`IsIconsView`. When adding a column or per-row behavior, it typically needs to be wired into all three templates that apply to it (details view has the column grid; icons/list are simpler). List and Icons use `Controls/VirtualizingWrapPanel.cs` to realize only visible/cache rows while preserving `ListBox` selection and DnD; do not replace it with a non-virtualizing wrap panel. `TabViewModel.IconSize` is a free (Finder-style) continuous zoom value — `ViewMode`'s named presets (ExtraLarge/Large/Medium) just set `IconSize` to a fixed number; don't reintroduce a fixed enum-keyed size map.

**Gallery mode is a fourth rendering mode layered on top of Icons view, not a separate view mode.** `TabViewModel.IsGalleryView => IsIconsView && IconSize >= 159.5` — it turns on when the size slider is pushed to the top, and `GalleryClose_Click` / Esc leave it by setting `IconSize` back down. Its layout is one `Grid` (`RowDefinitions="*,Auto,Auto"`) holding the large preview `Panel` (`x:Name="GalleryImageArea"`), a `Thumb.rowsplit` divider, and the horizontal filmstrip `ListBox` (`Classes="files gallerystrip"`). Things that are easy to break here:

- **Gallery hides chrome by collapsing rows, not just hiding elements.** The address-bar row, bookmarks bar and command bar each pair `IsVisible="{Binding !IsGalleryView}"` with an `Auto` row definition and an explicit `Height` on the element itself; if you give one of those rows a fixed height again, the chrome will leave an empty band in gallery mode. The vertical tab strip is also swapped: its tab `ListBox` binds `IsVisible="{Binding !SelectedTab.IsGalleryView}"` while the Exif panel binds the positive, so both live in the same grid row and only one shows.
- **The filmstrip height is window-scoped, not tab-scoped.** `MainWindowViewModel.GalleryStripHeight` (persisted in `AppSettings`, clamped 90–460) and the derived `GalleryThumbSize` (`GalleryStripHeight - 36`) are reached from inside the per-tab template via `$parent[Window].((vm:MainWindowViewModel)DataContext)`. Keep the drag clamp in `GalleryStripThumb_DragDelta`, the load-time validation, and the derived-size clamp consistent.
- **Wheel behavior is position-dependent** and handled in `OnPointerWheelChanged` before the Ctrl+zoom branch: over the filmstrip, vertical wheel is converted to horizontal scroll; over `GalleryImageArea`, wheel moves the selection via `TabViewModel.MoveGallerySelection`. Both are gated on "not Ctrl" so Ctrl+wheel still zooms out of gallery mode.
- **Background clicks must not clear the selection while in gallery**, because a cleared selection blanks the preview and reads as "the gallery closed". `TryStartMarqueeSelection` returns early for gallery mode, which is the single chokepoint covering all four call sites.

**Metadata for the gallery panel comes from the Windows property system.** `ImageMetadataService` opens an `IPropertyStore` via `SHGetPropertyStoreFromParsingName` and formats each value with propsys' `PSFormatForDisplay`, so display strings ("f/2.8", "1/125 秒", "4032 x 3024") are produced by Windows and already localized — don't hand-format PROPVARIANTs. The curated `PROPERTYKEY` list is the display order. `TabViewModel.UpdateGalleryMetadata` runs it off-thread with a `CancellationTokenSource` and re-checks `SelectedEntry`/`IsGalleryView` after the await, matching how `UpdatePreview` guards stale results.

**Keyboard shortcuts live in three handlers and which one fires depends on focus.** `Window.OnKeyDown` is the fallback that works when focus is anywhere in the window; `FileList_KeyDown` is attached to the file/filmstrip `ListBox`es and fires first when a row has focus; `GalleryImage_KeyDown` is attached to `GalleryImageArea` (which is `Focusable` and takes focus on click) so Del/F2/Ctrl+X/C/V still work when the user clicked the big preview instead of a row. Each of the two control-attached handlers sets `e.Handled` only for keys it actually consumes, so unhandled keys still reach `Window.OnKeyDown`. When adding a shortcut, decide deliberately which of the three it belongs in — adding it to only one is the usual cause of "the shortcut works in some places".

**A large amount of interaction logic is necessarily in `Views/MainWindow.axaml.cs` code-behind**, not pure XAML bindings — context menus that are built dynamically (shell menus, "New" submenu from `ShellNewService`, bookmarks bar nesting), drag-and-drop (both from-OS and to-OS), multi-key keyboard shortcuts, and Win32 handle lookups (`TryGetPlatformHandle()`) can't be expressed declaratively. Grep for the specific `_Click`/`_PointerReleased`/`_KeyDown` handler name before assuming logic doesn't exist yet.

**Feature definitions are catalog-backed.** `ContextActionCatalog` (tab scope only) feeds the tab context menu's "タブの詳細操作" group. Adding or renaming an action requires keeping the catalog entry, its dispatcher/handler in `MainWindow.axaml.cs` or `MainWindowViewModel`, and the catalog's startup validation in sync.

**Theme and accent color are not static resources** — `App.axaml` defines Light/Dark/OneDark `ThemeDictionaries` for the Chrome-ish chrome (tab strip bg, omnibox bg, etc.), and `ThemeService` pushes the live Windows accent color into Avalonia's `SystemAccentColor*` resources on startup and on every `PlatformSettings.ColorValuesChanged`. Don't hardcode colors in new views — use the `DynamicResource` keys already defined in `App.axaml` (`TabStripBg`, `ContentBg`, `TextPrimary`, etc.) so all themes, acrylic switching, and live accent-color changes keep working.

**Auto-update is Velopack**, wired the same way as the sibling project `Lhamiel` (`../Lhamiel` — check there first if extending update behavior, it's the reference implementation this was copied from). `Program.cs` runs `VelopackApp.Build().Run()` before the Avalonia app starts; `UpdateService.Check4Update` uses `SimpleWebSource` against the fixed Cloudflare R2 base URL and opens the download/apply dialog. `VelopackUpdateDialog`/`Velopack` are explicitly root-assemblies in the csproj (`TrimmerRootAssembly`) because their view models are only reached via compiled XAML bindings and would otherwise be trimmed by AOT.

**File-type icons are raster PNGs, not SVGs — this was a deliberate reversal, not the original design.** `Assets/MaterialIcons/` bundles [material-icon-theme](https://github.com/material-extensions/vscode-material-icon-theme) (MIT; see `NOTICE.md` there) as pre-rasterized 256×256 PNGs plus a `manifest.json` (extension/filename/folder-name and light-theme mappings → icon key, derived from the npm package's `dist/material-icons.json`). `MaterialIconService.ResolveIconKey` does the name→key lookup (exact filename → lowercase filename → extension → default), and `GetImage` loads/caches the PNG as an `IImage` through `AssetLoader` + `Bitmap`. **Do not swap this to runtime SVG rendering** (for example `Avalonia.Svg.Skia`): that route was tested and produced a fully black, never-composited window without a managed exception. If material-icon-theme's upstream icon set needs refreshing, the reproducible pipeline is: pull the npm tarball → use `dist/material-icons.json` plus `icons/*.svg` → rasterize with `resvg` → replace the PNGs and manifest in `Assets/MaterialIcons/`. The icon set is selected via `ShellOptions.IconSet` (`FileIconSet.Original` emoji / `Material` / `Windows`), persisted as the enum name string in `AppSettings.IconSet`; the legacy `AppSettings.UseMaterialIcons` bool is migrated on startup in `MainWindowViewModel` and then kept false — don't remove the old field, old settings.json files still carry it. The third set, `Windows`, renders the real Explorer shell icons: `ShellThumbnailService.TryGetIcon` (`SIIGBF_ICONONLY`) loads them lazily per visible row via `TabViewModel.EnsureWindowsIconAsync` (viewport/DataContext events in `MainWindow.axaml.cs`, plus a `SelectedTab`-change trigger so rows realized while the settings tab was frontmost get filled in). The sidebar (quick access, drives, recycle bin, recent files) follows the same setting through `RefreshSidebarAsync`, which precomputes an icon map off-thread; sidebar-owned shell `Bitmap`s are disposed on rebuild, while Material `IImage`s are shared with `MaterialIconService`'s cache and must never be disposed.

## Conventions specific to this repo

- All user-facing strings and code comments are Japanese (see any existing file). Keep new UI text and comments in Japanese to match.
- No dependency injection container — everything is constructed directly and wired via constructor params / shared instances passed down from `MainWindowViewModel`.
- For app-supplied helper commands (terminal, PowerShell, cmd, Explorer, editor), use `TrustedProcessLauncher` and argument lists instead of concatenating a command line. Direct `UseShellExecute` remains appropriate when opening the user-selected file itself through its registered shell handler.
- Avalonia 12's `WindowDecorations="BorderOnly"` is used deliberately so the custom-drawn caption buttons in the tab strip don't double up with OS-drawn ones — if captions ever look doubled, this is the property to check, not a bug in the custom buttons.
- `ApplicationIcon` (exe icon) and the `AvaloniaResource` `icon\app_icon.png` (window/taskbar icon) are two independent things that both need updating if the icon changes.
